Overview

Executive Summary: This project tests a microservices-based Payment Application focusing on Performance (JMeter) and Security (OWASP ZAP). We also verify API endpoints with Postman and ensure DB integrity in PostgreSQL.
Current Status: Ongoing – Additional scaling scenarios and advanced security checks planned for <>.

In this project, I tested a microservices-based Payment Application with a focus on Performance and Security. The application consists of:

  • Auth Service – for user registration and authentication
  • Payment Service – processes transactions, payments, and refunds
  • Order Service – manages orders and integrates with Payment Service
The goal was to ensure the system handles high user load and remains secure against common vulnerabilities.

Test Environment

Deployment: Docker Compose (3 separate containers for Auth, Payment, Order services)

Environment Details:

  • Auth Service: Node.js (Port 4001)
  • Payment Service: Java Spring Boot (Port 4002)
  • Order Service: Python Flask (Port 4003)
  • Database: PostgreSQL (Port 5432)
  • Reverse Proxy (Nginx): Single entry point on http://localhost:8080

Tools Used:

  • JMeter (Performance Testing)
  • OWASP ZAP (Security Testing)
  • Postman (API Testing)
  • DBeaver (DB verification)
  • GitHub Actions (CI/CD for test automation)

Key Functional Scenarios:

  • User Registration & Login (Auth Service)

Performance Testing

We used Apache JMeter to simulate concurrent users placing orders, making payments, and checking order status. The scenario included:

  1. Login (Auth Service)
  2. Create Order (Order Service) with random product data
  3. Pay (Payment Service) with valid/invalid card details
  4. Check Order Status (Order Service) to confirm success or failure

Load Profile:

  • 100 to 500 concurrent users over 5-minute ramp-up
  • Peak load of 500 users for 10 minutes
  • Spike test with sudden jump from 100 to 500 users in 30 seconds

Performance Metrics (Highlights):

  • Average Response Time: ~1.2s under 300 users, up to 2.5s at 500 users
  • 90th Percentile RT: 3.0s at peak load
  • Error Rate: ~2% at 500 users (mostly due to Payment timeouts)
  • Throughput: ~600 requests/second at peak

Analysis: The system remained stable up to 400 users with <1.5s average response time. Above 400, some Payment Service timeouts occurred. Further optimization needed in Payment Service’s DB queries.

Security Testing

We conducted a basic security assessment using OWASP ZAP to identify common vulnerabilities (OWASP Top 10). Key focus areas:

  • SQL Injection: Tested login forms and payment endpoints for injection vectors.
  • XSS (Cross-Site Scripting): Checked user input fields in the Order Service for improper sanitization.
  • Authentication & Session Management: Verified secure cookies (HttpOnly, Secure flags) and session timeout policies.
  • API Tokens: Payment requests require a valid JWT. Ensured expired tokens are rejected.

Findings:

  • Order Service (Low Risk): One reflected XSS alert in the search parameter, but no real exploit found. Mitigation: Proper URL encoding on the server side.
  • Payment Service (Medium Risk): Cookie flag missing “Secure” for session cookie on staging environment. Mitigation: Set Secure and HttpOnly flags in production build.
  • Auth Service (No High-Risk issues): All tokens validated properly, no injection vectors discovered.

Overall, the system was moderately secure; no critical vulnerabilities were identified, though minor improvements are needed in cookie configuration and input sanitization.

API Testing

For functional verification of REST endpoints (Auth, Payment, Order), we used Postman:

  • Auth Service: POST /api/auth/login, POST /api/auth/register
  • Payment Service: POST /api/payment/charge, POST /api/payment/refund
  • Order Service: POST /api/orders, GET /api/orders/{id}

We validated status codes (200, 201, 400, 401, etc.), JSON response structures, and negative scenarios (e.g., invalid tokens, insufficient account balance).

<>

DB Testing

Database: PostgreSQL running in a Docker container.

Key Checks:

  • Upon new user registration, user record is created in users table (Auth DB).
  • When an order is placed, an entry in orders table, referencing users.id.
  • Payments recorded in transactions table with correct order_id, status, and amount.

We ran SQL queries via DBeaver to ensure data integrity, foreign key constraints, and that rollback/commit behave correctly after a failed payment.

Regression

After each new feature or bug fix, we performed a partial regression on affected services (e.g., updating Payment logic triggers re-testing Payment + Order flows).

A smoke suite runs automatically on new Docker builds (Auth login, basic order creation). Periodically, a full regression covers all endpoints and cross-service integration scenarios.

<>

Automation

CI/CD: We set up GitHub Actions to run automated checks on each commit.

  • Functional Tests: Postman collections + Newman in the CI pipeline.
  • Performance Tests: JMeter scripts triggered via a separate Jenkins job (due to heavier resource usage).
  • Security Checks: OWASP ZAP baseline scan integrated in the nightly build.

Benefits: Immediate feedback on broken endpoints or performance regressions. We only deploy to staging if the pipeline is fully green.

Monitoring

Tools: Prometheus + Grafana for metrics, Kibana for logs

Key Metrics Tracked:

  • CPU / Memory usage of each container
  • Response Time / Error Rates from Payment and Order services
  • Number of successful vs failed transactions

Alerts are configured if error rates exceed 5% or CPU usage stays above 80% for more than 5 minutes. This helps promptly detect production incidents related to spikes in traffic or memory leaks.

Bug Reports
BUG-001: Payment Timeout at 400+ Users (Performance)

Status: Open

Priority: High

Description: At ~400 concurrent users, the Payment Service occasionally times out, returning 504 Gateway Timeout.

Steps to Reproduce:

  1. Run JMeter test with 400+ threads ramped up within 2 minutes.
  2. Observe Payment requests failing with 504 errors in the logs.

Expected Result: System should handle up to 500 users without timeouts or degrade gracefully.

Environment: Docker-based microservices (staging).

Attachments: JMeter Graph Screenshot

BUG-002: Missing Secure Flag on Payment Cookies (Security)

Status: Open

Priority: Medium

Description: Payment session cookies do not have the Secure attribute set on the staging environment.

Steps to Reproduce:

  1. Use OWASP ZAP to spider the Payment Service login flow.
  2. Check the “Set-Cookie” header in the response. The flag “Secure” is missing.

Expected Result: All session cookies must have “Secure” and “HttpOnly” set, especially in production.

Environment: Docker-based microservices (staging).

Attachments: OWASP ZAP Report Excerpt

Summary

Overall, the microservices handled up to ~400 concurrent users smoothly. Beyond that, Payment Service faced timeout issues (needs optimization or scaling).

Security scans revealed mostly low/medium vulnerabilities, such as missing Secure cookie flags and minor XSS vectors, but no critical flaws.

Future enhancements include load balancing for Payment Service, improved caching for Order Service, and stricter cookie policies in production.
